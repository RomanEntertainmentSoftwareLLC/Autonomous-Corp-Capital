# RPG v1 OPERATOR SUMMARY

## WHAT THE RPG SYSTEM WILL DO

### Core Purpose
Measure agent performance during live runtime work using XP, levels, and three active stats: Accuracy, Reliability, and Judgment. Speed is defined but dormant in v1.

### What Gets Measured (v1 Active)
- **Accuracy:** Correctness of work on first attempt
- **Reliability:** Consistency of execution over time  
- **Judgment:** Decision quality, escalation appropriateness, risk awareness

### What Is Defined But Dormant
- **Speed:** Task completion efficiency (requires timing data not yet captured)

### XP System
- Earn XP for verified successful actions
- Lose XP for documented failures
- Multipliers for first-try success and complex tasks
- Daily and per-session caps prevent farming

### Level Progression
- 10 levels (Initiate to Transcendent)
- XP thresholds: 0, 100, 200, 300... 900
- Stat minimums required for level advancement
- Anti-death-spiral protection (max -3 stat drop per session)

### Storage
- Per-agent files in `ai_agents_memory/{agent}/`
- `RPG_STATE.md` - Current stats and totals
- `RPG_HISTORY.md` - Session-by-session event log
- `RPG_ACHIEVEMENTS.md` - Unlocked milestones
- Simple markdown tables for human readability

---

## WHAT IT WILL NOT DO

### Not Implemented in v1
- **Speed stat** (defined but dormant until timing data exists)
- Real-time stat display during chat
- Automated proof verification (v1.1 goal)
- Cross-agent leaderboards
- Automatic level-up notifications
- Skill trees or unlockable abilities
- Integration with Axiom (comes later)

### Explicitly Excluded
- **No XP for chatter:** Idle conversation earns nothing
- **No self-generated tasks:** Cannot farm XP by creating busywork
- **No vanity edits:** Cosmetic-only changes don't count
- **No speed-running:** Rushed work loses XP, not gains it
- **No gaming the system:** Fake completions heavily penalized

### Out of Scope
- Historical backfill (starts from implementation date)
- Complex anti-cheat automation
- External integrations (Discord, web dashboards)
- Machine learning for pattern detection

## WHY AXIOM COMES LATER

### Axiom Should Inherit, Not Build

Axiom is the autonomous orchestration layer. It should:
- **Use** the RPG system
- **Not build** the RPG system

If Axiom has to create files, debug formulas, or establish processes, it cannot focus on its actual job: coordinating agents.

### Prerequisites for Axiom
Before Axiom touches RPG:
- Data files exist and are populated
- Formulas are tested and stable
- Process is established and working
- Edge cases are handled
- Operator is comfortable

### What Axiom Will Do
Once RPG is mature:
- Read agent stats for task assignment
- Consider levels when delegating
- Log its own activities for XP
- Use RPG data for coordination decisions

### What Axiom Will Not Do
- Create RPG files
- Debug XP calculations
- Establish approval workflows
- Fix data quality issues
- Handle formula edge cases

### Timeline
| Phase | Duration | Axiom Involvement |
|-------|----------|-------------------|
| 0-3 | 5 weeks | None |
| 4+ | Ongoing | Full integration |

Axiom starts using RPG after Week 5, not before.
